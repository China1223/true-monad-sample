# True Monad 项目核心概念界定文档

## 1. 概念体系概述

本文档旨在对 True Monad 系统架构中使用的所有核心概念进行明确定义和边界界定，确保项目参与者对系统设计哲学有统一的理解。

## 2. 核心概念详细界定

### 2.1 真元 (Prima / True Monad)

**定义**：人类输入的最原始、最不可分割的信息单元。真元只记录"人类说了什么"，不包含任何结构化信息。

**核心特征**：

- 原始性：保留人类输入的原始形态
- 不可分割性：每个真元都是独立的信息单元
- 无加工性：不包含AI的任何解释或转换

**物理载体**：`next/I-XXX.md`

**作用边界**：仅用于记录原始输入，不参与任何逻辑处理。

### 2.2 指令理解 (Instruction Understanding)

**定义**：AI对真元进行消化后，产出的"AI视角的理解文档"。

**核心特征**：

- 转换性：将原始真元转换为AI可处理的结构化理解
- 可追溯性：用于日后追溯"人类原始意图"与"AI实际理解"之间的偏差

**物理载体**：归档于`archive_instruction_understanding.csv`

**与真元的边界**：必须与真元分开存储，体现"真元与理解分离"原则。

### 2.3 交互三要素

#### Question (提问)

**定义**：AI在执行任务过程中，遇到不确定的、必须由人类拍板的信息盲区时发出的"求助信号"。

**核心特征**：

- 主动性：由AI主动发起
- 未决性：表示一个"过程性的、未决的"状态
- 交互性：需要人类介入才能解决

#### Advice (建议)

**定义**：AI基于自己的分析，主动向人类提出的方案或建议，等待人类批复。

**核心特征**：

- 主动性：AI主动提供
- 建议性：非强制性，需要人类确认
- 可选择性：人类可以选择采纳或拒绝

#### To Do (待办)

**定义**：需要人类在现实世界中配合完成的行动项。

**核心特征**：

- 行动导向：需要人类采取具体行动
- 外部性：通常涉及项目外部的执行
- 责任性：明确的人类责任项

**物理载体**：AI写入`current/`下的CSV，人类批复生成`next/Q/A/T-XXX.md`，最终归档于对应的`archive_*.csv`。

### 2.4 决策 (Decision)

**定义**：Question（或Advice）的"最终结果"与"结案状态"。

**核心特征**：

- 结论性：标志着某个不确定性已经被消除
- 已决性：是一个"结论性的、已决的"状态
- 里程碑性：是项目推进的里程碑

**与Question的边界**：Question是未决状态，Decision是已决状态，两者必须严格区分。

### 2.5 工作记录 (Work)

**定义**：AI在本轮中实际执行动作的结构化日志。

**核心特征**：

- 客观性：记录实际执行的动作
- 结构化：采用标准格式记录
- 可追溯性：便于后续审计和分析

**物理载体**：`current/current_work.csv`，归档于`archive_work.csv`。

### 2.6 认知环境 (Compressed Monad)

**定义**：对项目整体认知状态的结晶性描述文件，是整个项目的"大脑"。

**核心内涵**：

- 结晶性：对项目认知环境的完整描述
- 全局性：包含所有历史指令的结论性汇总
- 沉淀性：包含所有交互最终坍缩成的Decision沉淀
- 聚焦性：包含当前所处阶段的聚焦上下文

**关键特征**：

- 不归档性：不归入归档，由AI在每轮执行前进行原地更新
- 持续性：跨轮次持续更新的项目记忆

**物理载体**：`current/compressed_monad.md`

### 2.7 触发器 (Trigger)

**定义**：用于启动AI工作的纯信号字符。

**核心特征**：

- 纯信号性：白名单字符（`1`, `11`, `111`）
- 无语义性：AI绝对不要尝试分析其语义
- 直接性：看到触发器时直接扫描`next/`文件夹开始工作

### 2.8 文件即契约 (File as Pact)

**定义**：人与AI之间的一切沟通、状态流转，全部以物理文件为唯一真相来源。

**核心原则**：

- 唯一性：文件是唯一的真相来源
- 物理性：所有沟通都通过物理文件完成
- 对称性：输入输出都通过文件完成

### 2.9 真元与理解分离

**定义**：人类的原始意图（真元）与AI的消化结果（理解）必须物理隔离。

**实现方式**：

- 分开存储：真元和理解存放在不同的物理位置
- 分开归档：采用不同的归档机制
- 分开处理：不同的处理流程

### 2.10 Token 经济原则

**定义**：通过"逐级扫描"和"认知环境文件"机制，让AI在极低的Token消耗下恢复项目全貌。

**实现机制**：

- 逐级扫描：从认知环境到归档文件，按需逐层深入
- 认知环境：通过`compressed_monad.md`快速获取项目全局背景

### 2.11 职责绝对分离

**定义**：AI只负责内容的读写与逻辑思考；Python脚本负责数据的归档、校验、转译与清理；HTML工作台负责可视化交互。

**职责划分**：

- AI职责：读取指令和写入产出
- Python职责：归档和清理
- 工作台职责：可视化交互

### 2.12 Skill 纯粹性

**定义**：本系统的Skill仅关注人机协作的底层工作流。

**核心要求**：

- 关注点分离：具体的业务逻辑必须从Skill中彻底抽离
- 专门化处理：交由专门的子Skill或外部工具处理
- 底层化：只关注底层工作流机制

### 2.13 逐级扫描机制 (Progressive Scan)

**定义**：AI从`agent.md`进入系统后，为了在有限的Token预算内获取足够的上下文，必须严格执行的逐级扫描流程。

**扫描层级**：

1. 系统入口扫描
2. 上下文充足性判断
3. 认知环境扫描
4. 当前状态文件扫描
5. 归档与参考资料定向扫描

### 2.14 CSV格式强校验与转译机制

**定义**：在归档前对CSV文件进行语法和格式检查，以及对特殊字符进行转译处理的机制。

**核心要求**：

- 格式检查：确保列数、列名与归档模板一致
- 特殊字符转译：对换行符、逗号、双引号等进行转译处理
- 数据完整性：防止数据损坏和格式错误

## 3. 概念间关系图谱

### 3.1 核心概念关系

```
真元 → 指令理解 → 工作记录 → 决策
          ↓           ↓
       交互三要素 → 认知环境
```

### 3.2 系统流程关系

```
触发器 → 真元 → 指令理解 → 认知环境 → 工作记录 → 决策 → 归档
```

## 4. 设计原则总结

### 4.1 核心设计原则

1. **文件即契约**：文件是唯一的真相来源
2. **真元与理解分离**：原始意图与AI理解必须物理隔离
3. **输入输出对称**：`next/`进，`end/`出，边界清晰
4. **认知环境是项目大脑**：`compressed_monad.md`是项目认知状态的结晶
5. **Question与Decision的严格因果**：必须区分未决状态和已决状态

### 4.2 技术实现原则

1. **逐级扫描节省Token**：按需逐层深入，避免一次性加载所有历史
2. **AI只管读写，不管管理**：AI只负责内容读写，归档清理由Python脚本处理
3. **CSV转译与强校验**：确保归档数据的完整性和正确性
4. **触发器简化交互**：使用纯信号字符一键启动AI

## 5. 概念应用指南

### 5.1 概念使用规范

1. **严格遵循定义边界**：每个概念都有明确的定义和边界，不得混淆使用
2. **物理隔离要求**：不同概念必须通过物理文件隔离
3. **流程顺序要求**：概念的使用必须遵循系统设计的流程顺序

### 5.2 常见误区警示

1. **混淆Question与Decision**：严禁将未决状态与已决状态混淆
2. **混合真元与理解**：必须严格分离原始输入和AI理解
3. **忽视Token经济**：避免不必要的上下文加载，遵循逐级扫描机制
