# True Monad 项目本地运行架构：讨论记录、取舍与执行计划

## 1. 文档状态

- 日期：2026-06-23
- 状态：已确认并开始执行
- 范围：功能验证阶段的项目目录、运行代码所有权、Git 策略、未来 Skill 安装边界
- 暂不包含：正式 Skill/Plugin 发布、最终工作台视觉和完整交互模型

本文整理项目发起人与 Codex 在首版工作台完成后的多轮讨论。它既记录原始诉求的演进，也解释最终方案为何如此选择。

## 2. 原始需求与演进

### 2.1 首版需求

项目发起人首先要求根据四份根目录设计文档，实现一个功能完整优先的专注式工作台。UI 可以先保持基础，但必须能够查看认知环境、管理 Question/Advice/To Do、编辑指令、查看结果和历史，并完成归档。

首版据此实现了零第三方依赖的 Python Service、HTML 工作台、文件校验和 CSV 归档，并用一个最小 slave 轮次验证了指令、理解、Work、结果和归档回读。

### 2.2 Skill 规范问题

随后项目发起人指出：首版只是普通应用项目，不符合正式 Skill 的目录和运行方式；如果先完成全部功能再整体改造成 Skill，可能产生大量返工。

讨论结论是：不立即做正式发布包装，但先纠正运行边界，再在正确边界内完善功能。正式 Skill/Plugin 等功能稳定后再制作。

### 2.3 Service 与 Skill 版本一致性问题

项目发起人提出核心风险：如果 Skill 中修改了 Service，而项目使用的是复制到外部 `docs/` 的旧 Service，两者会长期不一致；如果项目持续运行全局 Skill 中的最新版脚本，新版本又可能与旧项目的数据结构不兼容。

讨论过的方案包括：

1. **始终运行全局 Skill 代码**：更新集中，但项目无法固定版本，其他机器和历史项目可能被新版破坏。
2. **把 Skill 作为 Git submodule**：能固定 commit，但项目仍依赖 Skill 目录存在，不符合“安装后脱离 Skill”的要求。
3. **项目本地固定安装**：首次安装时把完整 Service、脚本和契约写入项目；此后只运行项目本地版本。Skill 只负责首次安装、检测和用户明确授权的升级。

最终选择第三种方案。

### 2.4 可见目录与 Git 持久化

项目发起人要求使用可见的 `true-monad/`，而不是 `.true-monad/`。目的不是因为 Git 无法跟踪点目录——Git 实际可以跟踪 `.agents/`——而是让框架、Service 和历史在项目中清晰可见，并在 Skill 被删除后仍可工作。

项目的 True Monad 可见托管范围确定为：

- `AGENTS.md`
- `true-monad/`
- `docs/reference/`
- `docs/topic/`
- `docs/design/`

`.agents/skills/true-monad/` 将来只作为正式 Skill 的识别入口，不作为项目运行代码来源。

### 2.5 Git 中只保存完成状态

项目发起人接受使用 `true-monad/.gitignore` 排除 `next/current/end`，只将完成归档的历史以及框架代码、配置、文档纳入 Git。

这样可以减少未完成状态的分支冲突，但产生一个重要边界：被忽略的未归档数据无法通过 Git 回滚。因此升级必须在当前轮次归档后进行，或者先建立独立完整备份。

### 2.6 重复安装与升级

项目发起人明确要求：发现项目已经成功安装后，Skill 不得再次静默安装或覆盖。新版 Skill 与旧项目可能存在灾难性 Schema 不兼容。

最终规则：

- `true-monad/manifest.json` 是安装标志。
- 日常工作转交项目本地 `WORKFLOW.md` 和脚本。
- 再次请求安装时默认拒绝并警告。
- 升级是独立操作，必须由用户明确请求。
- 强制重装必须经过备份、兼容检查、staging 校验和失败恢复，不能直接复制覆盖。

## 3. 最终架构结论

True Monad 在功能验证阶段被定义为“由未来 Skill 安装和维护的项目级工作框架”。

```text
project-root/
├── AGENTS.md
├── true-monad/
│   ├── VERSION
│   ├── manifest.json
│   ├── WORKFLOW.md
│   ├── service/
│   ├── scripts/
│   ├── next/
│   ├── current/
│   ├── end/
│   ├── archive/
│   └── .gitignore
├── docs/
│   ├── design/
│   ├── reference/
│   └── topic/
└── tests/
```

运行时唯一权威来源是项目内 `true-monad/`。外部 Skill 升级不会自动改变已经安装项目的行为。

## 4. 为什么此前没有记录每轮用户原始指令

调查确认不是归档脚本丢失数据，而是指令根本没有进入 True Monad 文件入口。

### 4.1 聊天入口绕过 `next/`

原设计假定用户先通过工作台把指令写入 `next/I-XXX.md`，然后用触发器唤醒 AI。但此前的大多数需求直接通过 Codex 对话发送，没有任何桥接程序把聊天消息写入 `next/`。

### 4.2 `agent.md` 没有自动生效

首版使用根目录 `agent.md`。Codex 的仓库级持久指令入口是 `AGENTS.md`，因此 `agent.md` 只是普通文档，不会自动要求每轮先写原始指令。

### 4.3 工作契约本身缺少“聊天消息落盘”步骤

旧规则要求 AI“读取 next”，却没有要求收到聊天指令时先逐字写入 next。这使文件即契约只覆盖工作台输入，不覆盖直接对话。

### 4.4 Service 与当前 Codex 会话没有自动桥接

`trigger.signal` 只是待消费文件，没有监听器把当前聊天、Agent 执行和 Service 连接成一个事务。测试轮次之所以被记录，是因为主动通过工作台创建了 `I-001.md`。

### 4.5 修复方案

新的 `AGENTS.md` 要求：每次收到用户新任务，在实质工作前，先将用户原文逐字写入新的 `true-monad/next/I-YYYYMMDD-HHMMSS.md`。本轮重构将作为该规则启用后的首个真实记录轮次。

长期仍建议提供明确的会话桥接或工作台提交入口，避免完全依赖 AI 自律完成落盘。

## 5. 关键取舍

### 5.1 稳定性优先于自动升级

项目版本固定意味着安全、可复现和可回滚；代价是修复不会自动传播，必须显式升级。

### 5.2 项目独立性优先于全局单副本

每个项目拥有自己的已安装框架副本，能够脱离 Skill 工作；代价是不同项目可能停留在不同版本。这是有意的版本固定，不应被视为错误同步。

### 5.3 完成历史优先于未完成状态同步

Git 默认保存归档历史，不保存 `next/current/end`；代价是未归档状态需要额外备份，并且不能仅靠 Git 回滚恢复。

### 5.4 功能稳定优先于立即发布

当前先保证项目本地框架、Service、归档和工作台可执行。正式 Skill 和 Plugin 的 manifest、Marketplace、安装体验在功能稳定后再处理。

## 6. 本轮迁移计划

1. 保留四份根目录原始设计文档不变。
2. 把 Service、静态资源和脚本迁入 `true-monad/`。
3. 把 `next/current/end/archive` 迁入 `true-monad/`。
4. 建立 `VERSION`、manifest、WORKFLOW 和嵌套 `.gitignore`。
5. 将 `agent.md` 替换为正式 `AGENTS.md`，加入每轮原始指令落盘规则。
6. 建立 `docs/design`、`docs/reference` 和 `docs/topic` 的职责说明。
7. 更新测试与启动命令，确保端口和 API 保持兼容。
8. 记录并归档本轮真实指令，验证修复后的文件契约。
9. 使用自动测试和真实浏览器验证工作台核心路径。
