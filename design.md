# True Monad 系统架构与工作台设计文档

## 一、项目概述与命名

**项目名称**：True Monad（真元）

**核心设计哲学**：

1. **文件即契约（File as Pact）**：人与 AI 之间的一切沟通、状态流转，全部以物理文件为唯一真相来源。
2. **真元与理解分离**：人类的原始意图（真元）与 AI 的消化结果（理解）必须物理隔离。
3. **Token 经济原则**：通过"逐级扫描"和"认知环境文件"机制，让 AI 在极低的 Token 消耗下恢复项目全貌。
4. **职责绝对分离**：AI 只负责内容的读写与逻辑思考；Python 脚本负责数据的归档、校验、转译与清理；HTML 工作台负责可视化交互。
5. **Skill 纯粹性**：本系统的 Skill 仅关注人机协作的底层工作流。具体的业务逻辑（如原型设计、Git 版本管理、代码规范等）必须从 Skill 中彻底抽离，交由专门的子 Skill 或外部工具处理。

## 二、核心概念体系

### 2.1 真元（Prima / True Monad）

人类输入的最原始、最不可分割的信息单元。真元只记录"人类说了什么"，不包含任何结构化信息。

- **物理载体**：`next/I-XXX.md`

### 2.2 指令理解（Instruction Understanding）

AI 对真元进行消化后，产出的"AI 视角的理解文档"。它与真元分开存储，用于日后追溯"人类原始意图"与"AI 实际理解"之间的偏差。

- **物理载体**：归档于 `archive_instruction_understanding.csv`

### 2.3 交互三要素（Question / Advice / To Do）

这三个概念构成了 AI 与人类之间的"结构化沟通通道"：

- **Question（提问）**：AI 在执行任务过程中，遇到不确定的、必须由人类拍板的信息盲区时发出的"求助信号"。它是 AI 向人类发起的"未决状态"。
- **Advice（建议）**：AI 基于自己的分析，主动向人类提出的方案或建议，等待人类批复。
- **To Do（待办）**：需要人类在现实世界中配合完成的行动项。

**物理载体**：AI 写入 `current/` 下的 CSV，人类批复生成 `next/Q/A/T-XXX.md`，最终归档于对应的 `archive_*.csv`。

### 2.4 决策（Decision）——强调与 Question 的因果边界

**定义**：Decision 是 Question（或 Advice）的"最终结果"与"结案状态"。

**核心边界**：AI 必须严格区分 Question 和 Decision。Question 是一个"过程性的、未决的"状态，而 Decision 是一个"结论性的、已决的"状态。当人类对某个 Question 进行了批复，或者采纳了某个 Advice，这个交互就"坍缩"成了一个 Decision。Decision 是项目推进的里程碑，它标志着某个不确定性已经被消除。

### 2.5 工作记录（Work）

AI 在本轮中实际执行动作的结构化日志。

- **物理载体**：`current/current_work.csv`，归档于 `archive_work.csv`

### 2.6 认知环境（Compressed Monad）

**定义**：对项目整体认知状态的结晶性描述文件，是整个项目的"大脑"。

**核心内涵**：它不是对对话上下文的简单压缩，而是对项目认知环境的完整描述。它包含所有历史指令的结论性汇总、所有交互（Question/Advice）最终坍缩成的 Decision 沉淀，以及当前所处阶段（如原型、编程、维护）的聚焦上下文。

**关键特征**：不归入归档，由 AI 在每轮执行前根据新指令进行原地更新（Overwrite）。

- **物理载体**：`current/compressed_monad.md`

### 2.7 触发器（Trigger）

用于启动 AI 工作的纯信号字符（白名单：`1`, `11`, `111`）。AI 看到触发器时，绝对不要尝试分析其语义，直接扫描 `next/` 文件夹开始工作。

## 三、AI 启动流程：逐级扫描机制

### 3.1 第一层：系统入口（agent.md）

AI 每次启动必读。读取自身的角色定义、工作规则、文件读写规范及触发器白名单。

### 3.2 第二层：上下文充足性判断

AI 判断当前对话上下文是否已经包含了足够的历史项目信息。若充足，直接跳过后续扫描，进入工作状态；若不足，进入逐级扫描。

### 3.3 第三层：认知环境扫描（Compressed Monad Scan）

读取 `current/compressed_monad.md`。这是 Token 性价比最高的一层，AI 通过此文件快速获取项目的全局背景、历史决策沉淀及当前任务焦点。若读完此文件后上下文足够，则直接进入工作状态。

### 3.4 第四层：当前状态文件扫描（Current State Scan）

扫描 `current/` 目录下的 CSV 状态文件（`current_question.csv`, `current_advice.csv`, `current_todo.csv`, `current_work.csv`），了解当前有哪些悬而未决的事项。

### 3.5 第五层：归档与参考资料定向扫描（Archive & Reference Scan）

如果前四层仍无法满足需求，AI 才会按需、定向地扫描 `archive/` 中的历史轮次记录，以及 `refer/` 目录下的第三方参考资料。此层扫描完全由任务驱动，非必要不触发。

## 四、AI 每轮工作流与文件读写规范

### 4.1 更新认知环境

读取 `next/` 中的新指令后，首先更新 `current/compressed_monad.md`，将本轮的新认知融入全局与任务理解中。

### 4.2 读取真元

扫描 `next/` 文件夹，将所有文件（`I/Q/A/T-XXX.md`）作为本轮的完整指令。

### 4.3 产出指令理解

将 AI 对这批指令的理解写入独立的文件（供后续归档）。

### 4.4 执行与产出

执行任务，将本轮给人类看的所有结果以 Markdown 格式写入 `end/` 文件夹。

### 4.5 更新状态

将本轮产生的新 Question、Advice、To Do、Work 记录，写入 `current/` 下对应的 CSV 文件中。

## 五、归档与清理机制（Python 脚本职责）

### 5.1 CSV 格式强校验与转译机制

- **格式检查**：在归档前，必须对 `current/` 下的所有 CSV 文件进行语法和格式检查（列数、列名是否与归档模板一致）。若格式错误，立即中止归档并报错。
- **特殊字符转译**：由于 CSV 文件（特别是 `current_work.csv` 等）可能包含大模型生成的复杂文本，极易出现换行符、逗号、双引号等特殊字符破坏 CSV 结构。Python 脚本必须在归档前对 CSV 内容执行严格的转译（Escape）处理（例如将内容中的双引号转译为 `""`，将换行符包裹在引号内等），彻底降低数据损坏的风险。

### 5.2 归档真元与理解

将 `next/` 下的所有真元归档至 `archive_instructions.csv`；将 AI 的指令理解归档至 `archive_instruction_understanding.csv`。

### 5.3 归档物理产出

将 `end/` 文件夹下的所有 Markdown 文件内容，直接转换为 CSV 格式进行归档，不再保留独立的文件夹结构。

### 5.4 归档状态记录

将 `current/` 下的 CSV 状态文件，追加写入对应的 `archive_*.csv` 中。

### 5.5 环境清理

清空 `next/` 和 `end/` 文件夹；**保留** `current/compressed_monad.md`；清空或重置 `current/` 下的 CSV 状态文件。

## 六、工作台功能设计

### 6.1 基本信息

- **工作台名称**：True Monad 工作台
- **技术栈**：Python Web 应用
- **目录位置**：`docs/` 目录下
- **端口号**：8051
- **设计目标**：打造专注式工作室界面，提供高效的人机协作体验

### 6.2 核心功能模块

#### 6.2.1 认知环境查看

- 实时查看 `current/compressed_monad.md` 的内容
- 提供搜索和定位功能，方便快速查找关键信息
- 支持内容高亮和标记

#### 6.2.2 交互管理

- **Question 管理**：查看当前所有未解决的 Question，支持搜索、筛选和排序
- **Advice 管理**：查看当前所有待批复的 Advice，支持批量批复操作
- **To Do 管理**：查看当前所有待完成的 To Do 项，支持状态更新和优先级调整

#### 6.2.3 历史记录浏览

- **指令历史**：通过读取 `archive_instructions.csv`，展示所有历史指令记录
- **交互历史**：整合 `archive_questions.csv`、`archive_advice.csv`、`archive_todo.csv`，展示完整的人机交互历史
- **工作记录**：通过 `archive_work.csv` 查看 AI 的历史工作记录
- **轮次追溯**：支持按轮次查看历史状态，可追溯任意历史时刻的项目状态

#### 6.2.4 指令编辑与提交

- **多指令编辑**：支持同时编辑多个指令文件
- **实时保存**：编辑内容自动保存到 `next/` 目录下的临时文件
- **批量提交**：支持一次性提交多个指令文件
- **文件管理**：支持文件重命名、内容编辑、新增和删除操作
- **触发器集成**：内置触发器发送功能，提交指令后可一键触发 AI 工作

#### 6.2.5 状态监控

- **实时状态显示**：展示当前系统各组件的运行状态
- **进度跟踪**：监控 AI 工作进度和任务完成情况
- **异常告警**：及时发现并提示系统异常情况

### 6.3 工作流设计

1. **启动应用**：运行 `docs/server.py` 启动工作台服务
2. **访问界面**：通过浏览器访问 `http://localhost:8051` 打开工作台
3. **查看状态**：实时监控项目认知环境和交互状态
4. **编辑指令**：在工作台中编辑和管理待提交的指令文件
5. **提交触发**：提交指令后使用触发器启动 AI 处理
6. **监控结果**：实时查看 AI 处理过程和最终结果

## 七、项目目录结构全景

```
True Monad 项目根目录/
├── agent.md                    # 系统入口：AI 的岗位说明书与规则集
│
├── next/                       # 【输入口】真元与人类批复
│   ├── I-XXX.md                # 人类原始指令 (真元)
│   ├── Q-XXX.md                # 人类对 Question 的批复
│   ├── A-XXX.md                # 人类对 Advice 的批复
│   └── T-XXX.md                # 人类对 To Do 的回复
│
├── end/                        # 【输出口】AI 本轮的物理产出
│   ├── summary.md              # 总结
│   └── *.md                    # 其他详细产出
│
├── current/                    # 【状态区】AI 维护的中间状态与记忆
│   ├── compressed_monad.md     # 认知环境 (跨轮次记忆，不归入归档)
│   ├── current_question.csv    # 当前 Question 状态
│   ├── current_advice.csv      # 当前 Advice 状态
│   ├── current_todo.csv        # 当前 To Do 状态
│   └── current_work.csv        # 当前 Work 记录
│
├── archive/                    # 【历史区】Python 脚本维护的归档
│   ├── archive_instructions.csv
│   ├── archive_instruction_understanding.csv
│   ├── archive_questions.csv
│   ├── archive_advice.csv
│   ├── archive_todo.csv
│   ├── archive_work.csv
│   └── archive_end.csv         # 历史轮次 AI 产出的物理文件内容归档
│
├── refer/                      # 【知识库】第三方文档与参考资料
├── html/                       # 【交互层】HTML 工作台
└── docs/                       # 【管理层】子项目
    ├── scripts/                # Python 归档、校验与转译脚本
    └── server.py               # Python 后端服务 (端口: 8051)
```

## 八、设计原则总结

- **文件即契约**：人与 AI 之间的一切沟通都通过文件完成，文件是唯一的真相来源。
- **真元与理解分离**：人类说了什么（真元）和 AI 理解成什么（指令理解），必须分开存储、分开归档。
- **输入输出对称**：`next/` 进，`end/` 出，边界清晰。
- **认知环境是项目大脑**：`compressed_monad.md` 是项目认知状态的结晶。它不归入归档，是跨轮次持续更新的项目记忆。
- **Question 与 Decision 的严格因果**：Question 是未决的求助，Decision 是已决的结论。AI 严禁将两者混淆。
- **Skill 纯粹性**：Skill 仅关注底层工作流，原型设计、Git 管理等业务逻辑必须抽离。
- **逐级扫描节省 Token**：从认知环境到归档文件，按需逐层深入，避免一次性加载所有历史。
- **AI 只管读写，不管管理**：AI 只负责读取指令和写入产出，归档和清理全部交给 Python 脚本。
- **CSV 转译与强校验**：所有 CSV 文件在归档前必须经过特殊字符转译和格式检查，防止大模型生成的格式错误污染归档数据。
- **触发器简化交互**：`1`/`11`/`111` 作为纯触发信号，一键启动 AI。
- **专注式工作体验**：工作台设计遵循专注式工作室理念，提供高效、集中的协作环境。
